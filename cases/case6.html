Context

The Meta Dependency Problem

A significant portion of MMT GCC/Global's paid acquisition runs through Meta (Instagram + Facebook) — retargeting users who've searched for flights but haven't booked. While this channel drives volume, it creates three structural problems:

  • CPM inflation — Meta's auction gets more expensive every year
  • Attribution opacity — Meta's 7-day click / 1-day view windows inflate attributed revenue
  • No owned relationship — users who book via Meta ad stay as Meta's audience, not MMT's
Strategic Goal Use Meta as a discovery layer but capture users into MMT's owned lifecycle — reducing re-acquisition costs on subsequent bookings.

Diagnosis

Why Users Don't Convert Direct

📱

Friction at Handoff

Meta ad → app store → download → account creation → search is a 5-step funnel. Most users drop at step 3.

💬

No Value Hook

The landing experience after clicking a Meta ad doesn't give users a compelling reason to create an account vs just booking as guest.

🔔

Weak Re-engagement

Users who browse but don't book receive generic retargeting — not personalised nudges tied to their searched route and date.

🎁

No First-Party Incentive

There's no clear benefit to booking direct vs through Meta — no exclusive pricing, no loyalty point accrual, no saved preferences.

Strategy

Four-Part Conversion Architecture

01

Instant Web Booking (No-Download Path)

Build a frictionless PWA (Progressive Web App) landing page served directly from Meta ads. User lands, sees their searched route pre-filled, books in 3 taps — no app download, no account required. Email captured at booking confirmation.

02

First-Booking Incentive Layer

Users who create an MMT account during or post-booking receive: MMT Cash reward on next booking, saved itinerary, and price drop alert for their searched route. Makes account creation feel like a gain, not friction.

03

Personalised Re-engagement Lifecycle

Users who click Meta ad but don't book enter a 72-hour lifecycle: T+2hr WhatsApp nudge with exact route + price, T+24hr price movement alert, T+72hr last-call with dynamic urgency signal (seats filling). All triggered by Meta pixel event.

04

Second-Booking Direct Channel Lock-in

Post-first-booking, shift the user entirely to MMT-owned channels. Push notification + email sequence positions the MMT app as the default — with personalised deal alerts, trip history, and upcoming trip countdown.

Execution

Building the PWA Landing Page

Technical Architecture

  • Meta ad carries UTM + route parameters in URL
  • PWA pre-populates origin, destination, date from UTM
  • Search results rendered server-side for speed
  • Booking flow compressed to 3 screens: results → seat → payment
  • Email/phone captured at payment — account auto-created
Key Metric to Watch Meta ad click → PWA booking conversion rate. Baseline: ~2% (app store path). Target: 8–10% (direct PWA path).

Lifecycle Trigger Setup

  • Meta pixel fires on "search" event → enters WhatsApp journey
  • Route + price data passed via server-side API to lifecycle tool
  • Price movement tracked in real-time → triggers alert if fare changes ±5%
  • Urgency signal: "X seats left at this price" from inventory API
Stack Meta Pixel · WhatsApp Business API · Internal pricing API · Lifecycle automation (CleverTap / MoEngage)

Prioritisation

What to Build First

Phase 1 — Quick Win (4 weeks)

  • WhatsApp re-engagement lifecycle for browse abandoners
  • Low engineering effort, high conversion uplift
  • No new landing page needed — works with existing app deep links
  • Validates personalised nudge hypothesis before bigger investment

Phase 2 — Core Build (8 weeks)

  • PWA landing page with route pre-fill
  • Compressed 3-screen booking flow
  • Auto-account creation at payment

Phase 3 — Lock-in (ongoing)

  • First-booking MMT Cash reward programme
  • Second-booking direct channel incentives
  • Personalised deal alerts for searched routes
Why Phase 1 First The WhatsApp lifecycle requires the least engineering investment and gives us conversion signal data before we commit to the bigger PWA build. Ship, measure, then scale.

Risks

What Could Go Wrong

R1

Meta Policy Changes

Risk: Meta restricts pixel data sharing or changes attribution windows — breaking the trigger mechanism. Mitigation: Server-side pixel integration (Conversions API) is more resilient than browser pixel. Invest in this from day 1.

R2

WhatsApp Opt-out Rates

Risk: Aggressive 72-hour lifecycle triggers high opt-out rates, burning the channel. Mitigation: Cap at 2 messages per browse session. A/B test message frequency. Monitor opt-out rate as primary guardrail.

R3

PWA Performance on Low-End Devices

Risk: GCC users skew premium devices but India segments don't — PWA must load in <3s on 4G. Mitigation: Server-side rendering, aggressive caching, lazy-load non-critical assets. Performance budget defined before build starts.

Reflection

The Bigger Principle

🏠

Own Your Audience

Paid channels are rented audiences. Every growth strategy should have a parallel "own the user" track — email, WhatsApp, push — that reduces future acquisition cost for the same user.

⚡

Reduce Steps, Not Just Friction

The biggest conversion killer wasn't UX friction — it was the number of steps. From 5 steps (ad → store → download → signup → book) to 3 (ad → PWA → book) is a structural shift, not a UX polish.

📊

Attribution Drives Decisions

If you can't accurately attribute a booking to a channel, you can't optimise spend. Server-side attribution is infrastructure investment — it pays back in every budget conversation you have with marketing.
